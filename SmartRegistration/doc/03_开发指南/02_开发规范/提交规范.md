# 提交规范

## 1. 概述

本文档定义了智能挂号预约系统的代码提交规范，旨在确保代码提交信息的一致性和可追溯性，便于团队协作和代码审查。

## 2. 提交信息格式

### 2.1 基本格式

提交信息应遵循以下格式：

```
<类型>(<范围>): <描述>

<详细说明>

< footer >
```

### 2.2 格式说明

- **类型**：提交的类型，如 feat、fix、docs 等
- **范围**：提交影响的范围，如 auth、user、department 等
- **描述**：提交的简短描述，不超过50个字符
- **详细说明**：提交的详细描述，可多行
- **footer**：包含 Breaking Changes 或 Closed Issues 等信息

## 3. 提交类型

| 类型 | 描述 | 示例 |
|------|------|------|
| feat | 新功能 | feat(auth): 添加用户注册功能 |
| fix | 修复bug | fix(user): 修复用户信息更新失败问题 |
| docs | 文档变更 | docs: 更新API文档 |
| style | 代码风格 | style: 格式化代码 |
| refactor | 代码重构 | refactor(auth): 重构认证逻辑 |
| test | 测试相关 | test(user): 添加用户服务测试 |
| chore | 构建或依赖变更 | chore: 更新Maven依赖 |
| perf | 性能优化 | perf(database): 优化数据库查询 |
| ci | CI/CD配置 | ci: 配置Jenkins构建任务 |
| revert | 回滚提交 | revert: 回滚到之前的提交 |

## 4. 提交范围

提交范围应使用项目中的模块名称，如：

- auth：认证模块
- user：用户模块
- department：科室模块
- doctor：医生模块
- schedule：排班模块
- appointment：预约模块
- config：配置模块
- security：安全模块
- exception：异常处理模块
- util：工具模块
- middleware：中间件模块
- controller：控制器层
- service：服务层
- repository：数据访问层
- model：数据模型
- dto：数据传输对象

## 5. 提交描述

### 5.1 简短描述

- 以动词开头，使用现在时
- 首字母小写
- 不超过50个字符
- 清晰描述变更内容

**示例**：
- feat(auth): 添加用户注册功能
- fix(user): 修复用户信息更新失败问题
- docs: 更新API文档

### 5.2 详细说明

- 详细描述变更的原因和内容
- 可多行描述
- 提供足够的上下文信息
- 解释变更的影响

**示例**：

```
feat(auth): 添加用户注册功能

- 实现用户注册接口
- 添加用户信息验证
- 集成密码加密
- 完善错误处理

Closes #123
```

## 6. 提交规范

### 6.1 提交前检查

- 运行单元测试
- 检查代码风格
- 确保构建通过
- 检查提交信息格式

### 6.2 提交频率

- 每个功能或bug修复应作为单独的提交
- 不要将多个不相关的变更放在一个提交中
- 提交应小而频繁

### 6.3 提交信息语言

- 使用英语撰写提交信息
- 保持语言简洁明了
- 避免使用缩写和俚语

## 7. 提交示例

### 7.1 新功能

```
feat(auth): 添加用户注册功能

- 实现用户注册接口 POST /auth/register
- 添加用户信息验证
- 集成BCrypt密码加密
- 完善错误处理机制

Closes #123
```

### 7.2 修复bug

```
fix(user): 修复用户信息更新失败问题

- 修复PUT /users/{id}接口的参数验证
- 解决更新时的权限检查问题
- 添加详细的错误信息

Closes #456
```

### 7.3 文档更新

```
docs: 更新API文档

- 更新认证模块API文档
- 添加用户模块API文档
- 修正接口描述错误
```

### 7.4 代码重构

```
refactor(auth): 重构认证逻辑

- 提取认证核心逻辑到AuthService
- 优化JWT令牌生成和验证
- 简化认证过滤器
```

### 7.5 测试相关

```
test(user): 添加用户服务测试

- 为UserService添加单元测试
- 测试用户创建、更新和删除功能
- 模拟数据库操作
```

## 8. 分支管理

### 8.1 分支命名

- **master**：主分支，稳定版本
- **develop**：开发分支，集成功能
- **feature/xxx**：特性分支，开发新功能
- **bugfix/xxx**：bug修复分支，修复bug
- **release/xxx**：发布分支，准备发布
- **hotfix/xxx**：热修复分支，紧急修复

### 8.2 分支策略

1. 从develop分支创建feature分支
2. 完成开发后，将feature分支合并到develop
3. 从develop分支创建release分支
4. 完成测试后，将release分支合并到master和develop
5. 从master分支创建hotfix分支
6. 完成修复后，将hotfix分支合并到master和develop

## 9. 代码审查

### 9.1 提交前审查

- 自我审查代码
- 检查代码风格和质量
- 确保测试通过

### 9.2 代码审查流程

1. 提交代码到feature分支
2. 创建Pull Request
3. 团队成员进行代码审查
4. 解决审查意见
5. 合并到目标分支

### 9.3 审查重点

- 代码质量和风格
- 功能实现的正确性
- 性能和安全性
- 测试覆盖度
- 代码可读性

## 10. 最佳实践

- **保持提交小而专注**：每个提交只解决一个问题
- **写清晰的提交信息**：使用规范的提交信息格式
- **提交前测试**：确保代码通过测试
- **定期同步分支**：保持分支与主分支同步
- **使用描述性分支名称**：清晰表达分支的用途

## 11. 工具配置

### 11.1 Git Hooks

可以配置Git Hooks来自动检查提交信息格式：

1. **pre-commit**：提交前检查代码风格
2. **commit-msg**：检查提交信息格式

### 11.2 提交工具

- **Commitizen**：标准化提交信息
- **Gitmoji**：使用emoji丰富提交信息
- **Husky**：Git Hooks管理

## 12. 常见问题

### 12.1 提交信息格式错误

- **问题**：提交信息格式不符合规范
- **解决方案**：使用git commit --amend修改提交信息

### 12.2 提交内容过多

- **问题**：一个提交包含多个不相关的变更
- **解决方案**：使用git reset和git commit拆分提交

### 12.3 提交冲突

- **问题**：合并时发生冲突
- **解决方案**：手动解决冲突，然后提交

## 13. 总结

良好的提交规范是团队协作的重要基础，它可以提高代码的可维护性和可追溯性，便于代码审查和问题定位。通过遵循本文档的提交规范，团队可以保持代码仓库的整洁和有序，提高开发效率和代码质量。
