# 编码规范

## 1. 概述

本文档定义了智能挂号预约系统的编码规范，旨在确保代码的一致性、可读性和可维护性。所有开发人员在编写代码时应遵循本规范。

## 2. 命名规范

### 2.1 包名

- 使用小写字母，单词之间用点分隔
- 采用反向域名命名方式：`com.exclusive.blank.{模块名}`
- 示例：`com.exclusive.blank.controller`

### 2.2 类名

- 使用大驼峰命名法（PascalCase）
- 类名应清晰描述其职责
- 示例：`UserController`、`JwtTokenProvider`

### 2.3 方法名

- 使用小驼峰命名法（camelCase）
- 方法名应使用动词开头，清晰描述其功能
- 示例：`login()`、`getUserById()`

### 2.4 变量名

- 使用小驼峰命名法（camelCase）
- 变量名应清晰描述其用途
- 避免使用单个字母作为变量名（循环变量除外）
- 示例：`userId`、`appointmentTime`

### 2.5 常量名

- 使用全大写字母，单词之间用下划线分隔
- 示例：`MAX_REQUESTS`、`DEFAULT_EXPIRATION`

### 2.6 枚举名

- 使用大驼峰命名法（PascalCase）
- 枚举值使用全大写字母，单词之间用下划线分隔
- 示例：
  ```java
  public enum UserRole {
      ADMIN, DOCTOR, PATIENT
  }
  ```

## 3. 代码风格

### 3.1 缩进

- 使用4个空格进行缩进（不使用制表符）
- 代码块的左大括号放在行尾，右大括号单独占一行
- 示例：
  ```java
  if (condition) {
      // 代码块
  }
  ```

### 3.2 换行

- 每行代码长度不超过120个字符
- 长方法调用或复杂表达式应适当换行
- 示例：
  ```java
  userService.updateUser(
      userId,
      userName,
      userEmail
  );
  ```

### 3.3 空格

- 操作符两侧添加空格：`a + b` 而非 `a+b`
- 逗号后添加空格：`method(a, b, c)` 而非 `method(a,b,c)`
- 大括号内侧添加空格：`{ code }` 而非 `{code}`

### 3.4 空行

- 方法之间添加一个空行
- 逻辑块之间添加空行
- 类的成员变量和方法之间添加空行

## 4. 注释规范

### 4.1 类注释

- 使用Javadoc注释，描述类的用途、功能和注意事项
- 包含作者、创建日期等信息
- 示例：
  ```java
  /**
   * 用户控制器
   * 处理用户相关的HTTP请求
   * 
   * @author 技术团队
   * @since 1.0.0
   */
  @RestController
  @RequestMapping("/users")
  public class UserController {
  ```

### 4.2 方法注释

- 使用Javadoc注释，描述方法的功能、参数、返回值和异常
- 示例：
  ```java
  /**
   * 用户登录
   * 
   * @param request 登录请求参数
   * @return 包含令牌和用户信息的响应
   * @throws Exception 认证失败时抛出异常
   */
  @PostMapping("/login")
  public ResponseEntity<ApiResponse<Map<String, Object>>> login(@RequestBody LoginRequest request) {
  ```

### 4.3 代码注释

- 对复杂的业务逻辑添加注释
- 解释代码的设计意图
- 避免冗余注释（如注释显而易见的代码）

## 5. 异常处理

- 使用自定义异常类 `BusinessException` 处理业务异常
- 异常信息应清晰明了
- 在控制器层统一处理异常
- 示例：
  ```java
  if (user == null) {
      throw new BusinessException(ErrorCode.USER_NOT_FOUND, "用户不存在");
  }
  ```

## 6. 日志规范

- 使用SLF4J进行日志记录
- 日志级别：
  - `DEBUG`：调试信息
  - `INFO`：一般信息
  - `WARN`：警告信息
  - `ERROR`：错误信息
- 日志格式应包含时间、级别、类名和消息
- 示例：
  ```java
  private static final Logger logger = LoggerFactory.getLogger(UserService.class);
  
  logger.info("用户登录成功: {}", username);
  logger.error("登录失败: {}", e.getMessage(), e);
  ```

## 7. 安全规范

- 密码必须使用BCrypt加密存储
- 敏感信息不应硬编码在代码中
- 输入参数必须进行验证
- 避免SQL注入和XSS攻击
- 示例：
  ```java
  // 正确的做法
  user.setPassword(passwordEncoder.encode(password));
  
  // 错误的做法
  String sql = "SELECT * FROM users WHERE username = '" + username + "'";
  ```

## 8. 性能规范

- 使用批量操作减少数据库查询
- 避免在循环中执行数据库操作
- 使用缓存减少重复计算
- 合理使用索引优化数据库查询
- 示例：
  ```java
  // 批量保存
  userRepository.saveAll(userList);
  
  // 使用缓存
  @Cacheable("users")
  public User getUserById(Long id) {
      return userRepository.findById(id).orElse(null);
  }
  ```

## 9. 测试规范

- 为核心功能编写单元测试
- 测试覆盖率应达到80%以上
- 测试方法应清晰描述测试场景
- 示例：
  ```java
  @Test
  public void testLoginWithValidCredentials() {
      // 测试代码
  }
  ```

## 10. 版本控制

- 提交信息应清晰描述变更内容
- 遵循Git提交规范：
  - `feat`: 新功能
  - `fix`: 修复bug
  - `docs`: 文档变更
  - `style`: 代码风格
  - `refactor`: 代码重构
  - `test`: 测试相关
  - `chore`: 构建或依赖变更
- 示例：
  ```
  feat: 添加用户注册功能
  fix: 修复登录接口的异常处理
  docs: 更新API文档
  ```

## 11. 依赖管理

- 依赖版本应在pom.xml中统一管理
- 避免使用过时的依赖
- 定期更新依赖版本
- 示例：
  ```xml
  <properties>
      <java.version>17</java.version>
      <lombok.version>1.18.38</lombok.version>
  </properties>
  ```

## 12. 代码审查

- 代码提交前应进行自我审查
- 团队成员应相互审查代码
- 代码审查应关注：
  - 代码质量
  - 安全性
  - 性能
  - 可维护性

## 13. 最佳实践

- 遵循SOLID原则
- 保持方法职责单一
- 合理使用设计模式
- 避免代码重复
- 编写可测试的代码

## 14. 工具和插件

- 使用IntelliJ IDEA作为开发工具
- 启用代码风格检查
- 使用SonarQube进行代码质量分析
- 使用Lombok减少样板代码

## 15. 总结

本编码规范旨在提高代码质量和团队协作效率。所有开发人员应严格遵守本规范，确保代码的一致性和可维护性。规范将根据项目的发展和技术的进步不断更新和完善。
